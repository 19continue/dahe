# DaHeServeV2 `auth` 模块解读（最终版）
- 更新时间：2026-03-11
- 模块路径：`src/main/java/com/dahe/v2/modules/auth`
- 本文目标：说明后台鉴权、消息通知、小程序用户审核与并发保护在 `auth` 模块中的最终职责边界。

## 1. 模块职责
`auth` 目前承载四类核心职责：
1. 身份认证
- 后台管理员登录、会话获取、登出、会话续期。
- 小程序用户准入状态判断与登录准入协作。
2. 路由授权
- 后台按路由码做权限控制。
- 小程序按用户类型、审核状态、控制台能力做准入控制。
3. 审核治理
- 小程序用户审核、控制台开通、启停、回收与恢复都由 `auth` 模块统一编排。
4. 通知治理
- 审核结果通知、后台通知中心、消息任务投递、个人通知收件箱都统一放在 `auth` 模块。

## 2. 后台权限模型
1. 后台管理员只访问 `/api/v2/admin/**`。
2. 路由权限码统一使用路由路径本身，例如：
- `/users`
- `/users/recycle`
- `/assets/review`
- `/messages`
3. 角色权限来源于角色绑定的菜单码集合，不依赖硬编码角色名。
4. 超级管理员只有一个，具备全量后台能力，但不替代路由权限模型。

## 3. 小程序用户治理模型
1. 小程序用户不参与角色体系。
2. 小程序用户审核状态只有三类：
- `pending`
- `approved`
- `rejected`
3. 小程序控制台能力通过 `canConsole` 表示，不等价于后台角色。
4. 已通过的小程序用户支持进入回收站；回收后不可登录、不可继续被业务查询命中。
5. 已通过用户必须先回收，再允许彻底删除，防止业务数据被误删时失去追溯对象。

## 4. 小程序用户审核与回收链路
核心应用服务：`AdminUserManageServiceImpl`

### 4.1 审核页职责
路由：`/users`
- 只处理小程序用户审核。
- 支持通过、拒绝、开通控制台、启停、强制下线、移入回收站。
- 不再和回收治理混在同一页。

### 4.2 回收站职责
路由：`/users/recycle`
- 只展示已回收的小程序用户。
- 支持恢复、强制下线、彻底删除。

### 4.3 并发冲突保护
以下动作都要求前端提交 `expectedUpdatedAt`：
- 审核通过/拒绝
- 控制台开通/关闭
- 启用/禁用
- 移入回收站
- 恢复
- 删除

处理规则：
1. 后端先读取当前用户 `updatedAt`。
2. 若与前端提交的 `expectedUpdatedAt` 不一致，则直接拒绝操作。
3. 返回统一中文提示，避免多人同时处理同一条审核单时互相覆盖。

## 5. 通知中心与消息通知系统
### 5.1 设计目标
通知系统分为两层：
1. 个人通知
- 面向当前登录后台用户。
- 用于顶部“通知中心”展示与已读状态维护。
2. 消息任务
- 面向后台运营与治理。
- 用于创建一条“消息发送任务”，再异步派发给目标用户。

### 5.2 数据结构
1. `notice_task`
- 记录消息任务本身。
- 包含标题、内容、消息类型、发送对象、路由码、派发状态、投递结果。
2. `user_notice`
- 记录每个收件人的收件箱消息。
- 包含 `task_id`、`route_code`、`source_kind`、`sender_name` 等字段。

### 5.3 为什么拆成任务 + 收件箱
因为“创建消息”和“写入所有收件箱记录”不是一个量级：
- 创建消息必须尽快返回。
- 投递可能命中很多用户，不能阻塞业务请求。

所以当前链路是：
1. 业务或后台先落 `notice_task`。
2. `NoticeDispatchService` 异步解析收件人。
3. 异步批量写入 `user_notice`。
4. 更新任务状态、成功数、失败数。

### 5.4 异步派发实现
文件：`NoticeDispatchServiceImpl`
- 使用 `@Async("authNoticeExecutor")` 异步执行。
- 线程池配置放在 `AuthAsyncConfig`。
- 派发失败只影响消息任务状态，不阻塞原请求响应。

## 6. 消息发送对象与权限过滤
后台“消息通知”页支持多种发送对象：
1. `admin_all`
- 全部后台用户。
2. `admin_route`
- 按后台路由权限发送。
- 只有真正拥有该路由权限的后台用户才会收到。
3. `miniapp_approved`
- 全部已通过小程序用户。
4. `miniapp_console`
- 全部已开通控制台的小程序用户。
5. `explicit_users`
- 指定用户列表。

其中最关键的是 `admin_route`：
- 后端会读取管理员角色菜单权限。
- 仅当用户拥有目标路由码时，才会命中收件人列表。
- 这保证了“没有权限的人不能收到不属于他的治理消息”。

## 7. 后台通知中心与消息通知页分工
### 7.1 顶部通知中心
位置：后台主布局 `MainLayout`
- 展示当前用户个人通知。
- 支持筛选、分页加载、单条已读、全部已读。
- 点击通知时优先按 `routeCode` 跳转到对应页面。

### 7.2 消息通知页
路由：`/messages`
- 用于管理消息任务，而不是代替顶部通知中心。
- 支持分页查看、按类型/对象/状态筛选、创建任务、删除任务。
- 删除任务后，对应 `user_notice` 会同步软删除，不再继续展示给收件人。

## 8. 本轮新增/强化的接口
1. `GET /api/v2/admin/messages`
- 消息任务分页。
2. `POST /api/v2/admin/messages`
- 创建消息任务。
3. `DELETE /api/v2/admin/messages/{id}`
- 删除消息任务，要求 `expectedUpdatedAt`。
4. `PUT /api/v2/admin/auth/me/notices/read-all`
- 当前后台用户全部已读。
5. `POST /api/v2/admin/users/{id}/recycle`
- 小程序用户移入回收站。
6. `POST /api/v2/admin/users/{id}/restore`
- 小程序用户恢复。
7. `DELETE /api/v2/admin/users/{id}`
- 支持通过请求体提交 `expectedUpdatedAt`。

## 9. 设计约束
1. `auth` 模块不承载具体业务数据处理，只负责鉴权、审核、通知、会话治理。
2. 业务模块不得自己拼通知收件箱逻辑，应统一走 `UserNoticeService`。
3. 控制器只处理协议层与参数层，不在控制器中写审核编排。
4. 需要多人协作处理的审核类动作，必须继续走 `expectedUpdatedAt` 并发保护。

## 10. 当前剩余关注点
1. `auth` 模块下的治理能力已经较多，后续若继续扩容，可考虑把“消息任务管理”和“用户审核管理”再拆成更清晰的应用服务目录。
2. 当前通知派发已异步化，但如果未来发送量继续上升，应继续评估批量写入与任务重试策略。

## 11. 本轮新增能力
1. 用户审核消息的收件范围已扩展：
- 不仅推送给具备后台审核路由权限的管理员；
- 也会推送给具备控制台权限的小程序用户。

2. 通知投递仍保持异步，不阻塞审核等主业务请求响应。

3. 消息收件人判定继续遵守权限边界：
- 后台用户按路由权限判定；
- 小程序用户按 `canConsole=1`、`approved`、`enabled`、未回收等状态判定。

## 12. 本轮补齐
1. 消息发布已新增“消息保留配置”基础链路：
- 是否开启自动消息清理
- 消息在库中保留天数（天）
2. 当前仅完成配置读写与后台页面接入，自动清理任务本轮不实现。
3. 该配置统一收口到 `notice_dispatch_config`，后续真正实现自动清理时不再改消息任务主表结构。
