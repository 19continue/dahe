# DaHeServeV2 `assets` 模块解读（最终版）
- 更新时间：2026-03-11
- 模块路径：`src/main/java/com/dahe/v2/modules/assets`
- 本文目标：说明资源底库、资源审核、资源锁、回收治理与后台拆页后的最终结构。

## 1. 模块职责
`assets` 模块只做资源治理，不做业务编排。

当前职责包括：
1. 资源底库
- 保存上传文件的元数据、业务绑定关系、审核状态、回收状态。
2. 资源审核
- 对小程序上传资源做通过/驳回治理。
3. 资源锁
- 防止关键资源被误删。
4. 回收治理
- 支持恢复与彻底删除。
5. 目录治理
- 统一目录创建、占用检查、删除条件检查。

## 2. 资源审核模型
### 2.1 独立审核页
后台路由：`/assets/review`

设计目的：
- 不再让资源库页同时承担上传、编辑、审核、回收的全部职责。
- 审核动作集中到独立页面处理，减少误操作。

### 2.2 审核状态
- `pending`：待审核
- `approved`：已通过
- `rejected`：已驳回

### 2.3 驳回处理
驳回不是简单改状态，而是：
1. 写入驳回原因。
2. 将资源自动移入回收站。
3. 保留记录，避免误审导致资源直接消失。

## 3. 资源锁模型
### 3.1 设计目标
防止品牌主图、默认占位图、长期引用图等关键资源被普通管理员误删。

### 3.2 行为规则
1. 只有超级管理员可以：
- 设置资源锁
- 关闭资源锁
- 重置资源锁密码
2. 普通管理员删除锁定资源时：
- 必须输入正确密码解锁
3. 密码只存摘要，不存明文。
4. 超级管理员忘记密码后可直接重设，不要求原密码。

### 3.3 作用范围
资源锁作用于：
- 移入回收站
- 回收站彻底删除

## 4. 回收治理
### 4.1 正常资源页
路由：`/assets/library`
- 只处理正常资源。
- 支持编辑、审核、锁定、排序。

### 4.2 回收站页
路由：`/assets/recycle`
- 只处理已回收资源。
- 支持恢复、彻底删除。
- 锁定资源彻底删除前仍需密码解锁。

### 4.3 目录治理页
路由：`/assets/folders`
- 只处理目录创建、占用和删除。

## 5. 后台页面拆分结果
后台资源治理已拆为四页：
1. `/assets`
- 资源总览
2. `/assets/library`
- 资源库
3. `/assets/review`
- 资源审核
4. `/assets/recycle`
- 回收站
5. `/assets/folders`
- 目录治理

页面拆开后，治理动作的边界更清晰：
- 总览页看状态与入口
- 审核页专做审核
- 资源库专做管理
- 回收站专做恢复与销毁
- 目录页专做结构治理

## 6. 并发冲突保护
资源审核已支持 `expectedUpdatedAt`。

处理规则：
1. 前端提交审核动作时带上当前行的 `updatedAt`。
2. 后端审核前先校验数据库中的 `updatedAt` 是否一致。
3. 不一致则拒绝操作，避免多人同时审核同一资源时后写覆盖先写。

## 7. 核心接口
1. `GET /api/v2/admin/assets`
- 资源分页
- 已支持 `lockedFlag` 等筛选
2. `PUT /api/v2/admin/assets/{id}/review`
- 资源审核
- 驳回时自动回收
3. `PUT /api/v2/admin/assets/{id}/lock`
- 设置、关闭、重置资源锁
4. `DELETE /api/v2/admin/assets/{id}`
- 移入回收站
- 锁定资源支持 `unlockPassword`
5. `DELETE /api/v2/admin/assets/{id}/purge`
- 彻底删除
- 锁定资源支持 `unlockPassword`
6. `POST /api/v2/admin/assets/{id}/restore`
- 资源恢复

## 8. 设计约束
1. `assets` 只处理资源治理，不负责业务模块自己的流程编排。
2. 小程序专属接口继续放在 `/api/v2/miniapp/**`，不再和后台共享路径混写。
3. 资源审核、回收、锁定都应继续通过统一后台路由权限控制，不依赖写死角色名。

## 9. 当前剩余关注点
1. `MediaAssetController` 仍然偏大，后续应继续按治理场景拆控制器。
2. 小程序公共运营资源映射层还未正式引入；当前 `assets` 模块解决的是底库与治理，不是“小程序页面公共资源编排”。

## 10. 本轮新增能力
1. 新增 `media_asset_reference` 引用表。
- 用于持久化记录资源当前被哪些模块引用。
- 删除资源前可直接查询引用明细，而不是依赖页面临时推断。

2. 资源删除前提示已升级：
- 若资源存在引用，后台删除确认会展示“该资源已被以下模块使用，删除后会影响以下模块的功能，确定删除吗？”
- 同时按模块分组展示田块、农事记录、种子批次、导出模板、用户等业务对象的简略信息。
- 摘要只展示可读业务信息，不再把 `ID` 直接暴露给管理员。

3. 回收站彻底删除已升级为“双删”：
- 删除服务器本地文件。
- 物理删除 `media_asset` 数据库记录。
- 同时物理删除 `media_asset_reference` 中对应的引用快照记录。

4. 资源总览页已去掉“业务引用概览”区块，资源治理信息收口到资源库删除确认与引用详情接口。

## 11. 本轮补齐
1. 资源删除前的引用确认已改为分页模式：
- 首次最多加载 5 条引用
- 若仍有剩余，后台显示“点击加载更多”
- 引用信息按模块分组展示，不一次性灌满全部列表
2. 资源引用不再只覆盖农事记录：
- 田块封面改为显式记录 `coverImageAssetId` 并在增改删时同步绑定/解绑
- 企业信息、企业产品、企业荣誉、作物分类、作物品种等“只存图片 URL”的模块，统一通过资源绑定协调服务回写 `media_asset_reference`
3. 回收站彻底删除资源时，会同步物理删除：
- 本地文件
- `media_asset` 记录
- `media_asset_reference` 引用快照
4. 后台资源总览页继续不展示“业务引用概览”，引用信息只在删除确认与引用详情接口中使用。

## 12. 小程序静态资源
`assets` 模块现在额外承接一条“固定 URL 的小程序静态资源”链路，但它不再落到 `media_asset` 表，而是独立使用 `miniapp_static_asset`。

设计原因：
1. 这类资源不是业务附件，不应混入普通资源库、审核流、回收站。
2. 这类资源要求可控 URL，因此需要显式保存“文件名主体”，而不是继续沿用上传时生成随机文件名。
3. 权限必须收口到超级管理员，不能通过普通角色配置放开。

约束：
1. 仅超级管理员可见、可上传、可编辑、可删除。
2. 文件固定上传到服务器真实目录 `/assets`，对外静态访问路径为 `/assets/**`。
3. 上传时必须指定“保存文件名称（不含后缀）”，系统根据内容识别真实后缀并拼出最终 URL。
4. 不设回收站，删除即同时删除数据库记录和服务器文件。
5. 删除时必须输入当前超级管理员登录密码，不复用普通资源锁密码。

核心表：
1. `miniapp_static_asset`
- `display_name`：后台查看用的展示名称
- `storage_name`：固定 URL 使用的文件名主体
- `file_ext`：真实后缀
- `file_url`：最终访问地址
- `remark`：补充说明

核心接口：
1. `GET /api/v2/admin/miniapp-static-assets`
- 小程序静态资源分页
2. `POST /api/v2/admin/miniapp-static-assets/upload`
- 上传静态资源
3. `PUT /api/v2/admin/miniapp-static-assets/{id}`
- 编辑展示名称和备注
4. `DELETE /api/v2/admin/miniapp-static-assets/{id}`
- 永久删除静态资源

## 13. 小程序用户上传资源引用追踪补齐
此前普通业务图片已经能通过 `bindAssetsToBiz(...)` 维护引用，但“小程序用户头像”这类只存 URL 的链路没有被纳入引用快照。

本轮已补齐：
1. 小程序用户登录/申请写资料时，会按 `auth + userId` 同步头像资源引用。
2. 小程序用户主动更新头像时，也会同步重建头像资源引用。
3. 启动自举会回填历史 `user.avatar_url / user.wx_avatar_url` 对应的资源引用快照。

结果：
1. 用户上传过的头像资源不再因为“看起来没有业务绑定”而被误删。
2. 资源删除确认里，`auth` 模块下的用户引用能够被正常识别与展示。
