# DaHeServeV2 `company` 待改进的点（最终版）
- 更新时间：2026-03-05
- 对应解读文档：`docs/backend-module-review/modules/company-模块解读.md`

## P0（高优先级）
1. company_info 单例约束仍主要依赖应用层
- 现状：已在写服务中做“保存后清理重复行”，但数据库层未加硬约束。
- 风险：绕过服务层写库时可能再次出现多行并存。
- 建议：补充数据库层单例约束方案（迁移脚本 + 启动自举兼容）。

2. company 变更审计粒度仍可细化
- 现状：依赖通用操作日志，字段级快照对比能力不足。
- 建议：在 `oplog` 增加 company 关键字段快照与变更差异记录。

## P1（中优先级）
1. 排序冲突治理可继续增强
- 现状：已做默认排序与非负校验，但未提供重排接口。
- 建议：补充重排 API，减少运营端频繁手工改 sort_order 的成本。

2. 写服务测试覆盖不足
- 现状：缺少针对长度校验、URL 校验、联系方式格式校验、单例收敛的自动化测试。
- 建议：补齐 command service 单测与接口集成测试。

## P2（低优先级）
1. 公开端可增加短时缓存
- 现状：公开接口每次实时聚合查询。
- 建议：结合发布频率增加缓存，提高高并发下稳定性。

2. 富文本引入前置安全策略
- 现状：当前字段以纯文本为主。
- 建议：若后续引入富文本，需同步 XSS 清洗与白名单策略。
