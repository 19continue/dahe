# DaHeServeV2 `user` 模块待改进的点（总版）

- 更新时间：2026-03-03
- 对应解读文档：`docs/backend-module-review/modules/user-模块解读.md`
- 目标：识别用户主数据域在一致性、性能与可维护性上的风险点，并给出可落地改进方向。

## P0（建议优先处理）

1. 用户写入规则统一收口（已完成）
- 落地：新增 `UserDomainService`，将小程序登录建档、审核态迁移、角色/控制台变更、启停、后台账号创建等关键写操作收口到 `user` 域。
- 效果：`AuthPortalServiceImpl` 与 `AdminUserManageServiceImpl` 已改为调用领域写服务，不再在上层散写关键字段。

2. 状态字段与类型字段常量化（部分完成）
- 落地：新增 `UserDomainConstants`，并让 `AuthDomainConstants` 对应值引用 user 域常量。
- 剩余：其他历史业务模块（非 auth/user）仍有零散字面量，需按批次继续收敛。

3. `findByWxOpenId` 重复数据告警（已完成）
- 落地：查询改为 `limit 2` 检测重复；发现重复时记录错误日志并补充重复数量。
- 现状：保持兼容返回首条记录，不影响线上行为。

## P1（建议近期处理）

1. 分页查询关键字检索缺少精细化策略
- 现状：`keyword` 对 `nickName/realName/phone/wxOpenId` 全量 `like`。
- 风险：大数据量下索引利用率下降，分页成本上升。
- 建议：按字段类型拆分策略（手机号、openId 走前缀/精确匹配，昵称实名走模糊）。

2. 分页排序维度单一
- 现状：仅 `created_at desc`。
- 风险：同时间戳数据较多时，翻页稳定性较弱。
- 建议：增加二级排序键（如 `id desc`）保证稳定分页。

3. 查询参数边界未在服务层二次保护
- 现状：`pageSize` 上限主要依赖控制器约束。
- 风险：内部调用绕过控制器时可能触发超大分页。
- 建议：在服务层增加分页参数兜底（最小值、最大值、默认值）。

## P2（可排期优化）

1. 用户域自动化测试不足
- 现状：`openId` 查找、分页组合筛选、逻辑删除隔离缺少系统化测试说明。
- 建议：补齐 `service` 层单测与集成测试矩阵。

2. 用户领域对象可进一步类型化
- 现状：`status/userType/avatarSource` 使用字符串。
- 建议：逐步引入值对象或枚举映射，降低维护成本。

3. 读性能可按调用热点优化
- 现状：鉴权链路高频按 `id` 读用户。
- 建议：在保证一致性的前提下评估短时缓存或轻量投影查询。

## 建议实施节奏

1. P0 已完成 2.5 项：
- 已完成：写规则收口、重复数据告警；
- 部分完成：常量化（auth/user 已收敛，其他模块待推进）。
2. 下一步优先 P1：
- 关键字检索策略分层；
- 分页参数服务层兜底（上限、默认值）；
- 继续收敛其余模块中的用户状态/类型字面量。
3. 最后处理 P2：完善测试矩阵与领域类型化演进。
