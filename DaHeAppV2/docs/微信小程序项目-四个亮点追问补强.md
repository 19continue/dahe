# 微信小程序项目四个亮点追问补强

这份文档只写“面试官继续追问时你怎么接”。回答风格按校招 / 初级后端去写，重点是稳，不是炫。

## 1. 先认证、后登录的准入机制

### 问：为什么要做先认证、后登录，而不是用户一进来就登录？

答：因为这个项目不是纯开放型工具，而是有业务准入要求。微信 `openid` 只能证明“这是同一个微信用户”，不能证明“这个人已经有系统使用资格”。所以我把流程拆成两段，先根据 `openid` 查认证状态，再决定是否签发 session。这样能把审核中、未通过、停用、资格收回这些状态都拦在登录之前。

### 问：为什么不让未通过用户也先登录，再靠页面权限限制？

答：那样会把“准入”推迟到业务层，风险更大。用户一旦先拿到登录态，后续每个页面、每个接口都要自己再判断是否放行，容易漏。现在是把校验收口在 session 签发前，只有允许进入系统的用户才发 token，边界更清晰。

### 问：为什么 `validate` 的时候还要再看用户状态？

答：因为用户状态是会变化的。比如一个用户昨天还是 approved，今天可能被停用或者资格被收回。如果 `validate` 只校验 token 本身，不再校验用户当前状态，那旧 token 还能继续用。所以这里做的是“双重校验”：既看 session 是否有效，也看用户当前是否仍允许登录。

### 问：审核中、未通过、已停用、资格已收回有什么区别？

答：

- 审核中：资料已提交，但管理员还没审批，不允许登录。
- 未通过：管理员已驳回，用户可以修改资料后重新提交。
- 已停用：账号被禁用，不允许继续使用，也不应该再发登录态。
- 资格已收回：之前可能是可用用户，但业务资格被撤销，需要重新申请。

### 问：你这里的核心亮点到底是什么？

答：我觉得核心不是“做了个登录页”，而是把微信身份识别、业务认证状态和 session 签发拆开了。这样系统既能支持认证流程，又能把登录放行规则集中管理。

## 2. Redis session 缓存 + bypass + 高德缓存

### 问：为什么 Redis 只做 session 读缓存，不直接把 session 全放 Redis？

答：当前实现里数据库是最终真相。这样做的好处是，session 状态有持久化记录，失效、审计和追踪更稳定。Redis 主要解决高频读取带来的性能问题。如果 Redis 出问题，系统还能直接退回数据库，不至于整个鉴权链路不可用。

### 问：Redis bypass 是什么意思？

答：就是 Redis 发生异常后，不让每个请求继续反复踩 Redis，而是在一小段时间里直接跳过 Redis，回数据库查 session。项目里是通过记录一个 `redisBypassUntilAt` 时间戳控制的，只要当前还在 bypass 窗口，就先不走 Redis。

### 问：为什么不每次 Redis 出错都继续重试？

答：如果 Redis 已经明显不稳定了，每次请求都先重试 Redis，只会把一次故障放大成所有请求都变慢。bypass 的目的就是止损，让系统快速降级，优先保主业务可用。

### 问：那 bypass 的问题是什么？

答：问题是数据库压力会上来，所以它只是短暂降级，不是长期方案。适合当前项目这种中小体量场景。如果后续量更大，就要进一步考虑 Redis 高可用、监控告警和更细的降级策略。

### 问：高德接口为什么要做缓存？

答：因为这是第三方接口，存在网络时延、调用额度和 QPS 限制。同一个地区、同一块田附近的位置和天气，在短时间内变化不大，直接缓存性价比很高。这样既能减少真实请求次数，也能提升首页和田块页的响应速度。

### 问：位置缓存为什么不是直接用原始经纬度做 key？

答：原始经纬度太细了，命中率会很差。项目里做了网格化，把一定范围内的经纬度折叠到同一个缓存 key。这样“附近田块展示”和“当前位置匹配”这些场景就能复用缓存。

### 问：本地缓存和 Redis 缓存分别承担什么角色？

答：Redis 开启时，主要用 Redis 做跨请求共享缓存；如果 Redis 关闭或者不想依赖 Redis，则退回本地有界缓存。它更像一个可切换的工程方案，而不是复杂的多级缓存中台。

## 3. 农事记录步骤驱动 + schema + extraJson

### 问：为什么不用固定表字段把所有农事数据都存下来？

答：因为不同农事步骤差异很大。如果把所有步骤可能用到的字段都塞进主表，表会越来越胖，而且大部分字段在很多记录里都是空的。项目里把公共字段保留在主表，把步骤特有字段放进 `extraJson`，灵活性更高。

### 问：那为什么又需要 schema？

答：因为只有 `extraJson` 还不够。没有 schema 的话，前端不知道该渲染哪些字段，后端也没法校验提交的 JSON 是否合法。schema 相当于这份动态数据的合同，规定了 key、label、类型、是否必填和选项范围。

### 问：为什么说这是“步骤驱动”而不是“自由扩展”？

答：因为动态字段不是让用户随便加，而是由当前农事步骤决定。先确定 `stepId`，再解析出对应 schema，最后用这份 schema 去校验 `extraJson`。这样同一种步骤的记录结构会比较稳定，方便维护和展示。

### 问：`extraJson` 的缺点是什么？

答：灵活是它的优点，但它也有缺点，比如：

- 按某个扩展字段做复杂查询不如结构化字段方便
- 约束更多依赖应用层 schema 校验
- 如果 schema 变更不谨慎，历史数据兼容会变复杂

所以它适合扩展字段，不适合把所有关键查询条件都放进去。

### 问：为什么列表展示时还要解析 labelMap？

答：因为 `extraJson` 里的 key 往往是机器字段名，select 里存的也可能是 value code。直接返回给前端不够友好。所以后端会把 schema 再解析一遍，生成字段 label 和选项 label 映射，让前端能更轻松地展示为中文语义。

## 4. `@Async + ThreadPoolTaskExecutor` 异步通知

### 问：为什么这里用 `@Async`，不用 MQ？

答：当前项目通知量级不算特别大，主要需求是不要阻塞认证申请、审核、状态变更这些主请求。在这个阶段，用 `@Async + ThreadPoolTaskExecutor` 可以用很低的成本把通知逻辑从主链路里拆出来，维护复杂度也比 MQ 低，所以是更务实的选择。

### 问：`@Async` 的核心价值是什么？

答：核心价值不是“异步”这两个字，而是把主请求和通知派发拆开。主请求只要把任务落库即可返回，后面的收件人解析和批量写通知由异步线程处理，这样用户感知到的接口耗时会明显更稳定。

### 问：为什么一定要配专用线程池？

答：如果直接用默认线程池，不好控制并发和队列，也不方便定位问题。这里单独配置 `authNoticeExecutor`，是为了把认证通知这类异步任务隔离出来，避免互相影响，同时也更方便调优和排查。

### 问：怎么避免一个通知任务被重复派发？

答：异步方法开始时会先尝试把 `notice_task` 的状态从 `pending/failed` 更新成 `sending`。只有抢占成功的线程才继续往下处理。这样能避免同一任务被多个线程重复消费。

### 问：这种方案的边界在哪里？

答：它更适合单体应用内的轻量异步，不等于消息中间件。优点是实现简单；边界是跨进程可靠投递、超大流量削峰和更强的消息堆积处理能力都不如 MQ。如果未来通知量继续增长，可以把 `notice_task` 当作演进入口，再逐步迁到 MQ。

## 这四个点最容易被问穿的地方

### 1. 说不清“状态”和“登录态”的区别

补法：记住一句话，状态是业务准入条件，登录态是技术会话凭证，状态通过后才发登录态。

### 2. 把 Redis 说成 session 真相来源

补法：记住一句话，数据库是最终真相，Redis 只是热点读取加速层。

### 3. 把 `extraJson` 吹成万能动态平台

补法：记住一句话，它解决的是“步骤扩展字段”问题，不是完整低代码引擎。

### 4. 把 `@Async` 吹成 MQ 架构

补法：记住一句话，当前是轻量异步派发，目标是解耦主链路，不是消息平台建设。

## 你现在最该背熟的 8 句话

1. 小程序登录链路里，微信身份识别和业务准入是分开的。
2. 只有认证状态允许时，后端才签发 session。
3. `validate` 不只验 token，还会再次验用户当前状态。
4. session 查询优先走 Redis，数据库仍是最终真相。
5. Redis 出现异常时会短暂 bypass，避免把故障扩散到所有请求。
6. 高德缓存的主要目标是降低真实接口调用成本和响应时延。
7. 农事记录采用固定字段加 `extraJson` 的方式承载步骤扩展字段。
8. 通知链路采用 `@Async + ThreadPoolTaskExecutor` 做轻量异步，避免主请求被批量通知拖慢。
