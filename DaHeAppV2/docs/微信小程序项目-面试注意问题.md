# 大禾种业微信小程序项目面试注意问题

## 1. 先定项目定位

面试时你应该把这个项目定位成：

一个面向农业种业场景的业务管理系统，包含微信小程序、后台管理端和 Spring Boot 后端，重点解决认证准入、田块管理、农事记录、种子质量追溯、资源上传和后台治理问题。

不要把它讲成：

1. 高并发互联网平台
2. 微服务票务系统
3. 分布式中间件平台
4. 复杂交易系统

## 2. 你必须先记住的几个真实点

### 2.1 项目架构

1. 是模块化单体，不是微服务。
2. 有前后端分离，但不是 Spring Cloud 全家桶。
3. 小程序接口和后台接口通过路由域严格隔离。

### 2.2 鉴权

1. 不是纯 JWT 方案。
2. 更准确是：`token session + Redis 会话缓存 + 自定义鉴权拦截器 + 路由授权服务`。
3. Spring Security 在这里更像基础安全能力和密码编码器入口，不是完整权限链主角。

### 2.3 Redis

1. 主要用于 token session 读取加速。
2. 还用于高德位置天气缓存。
3. 不要说成你做了全站缓存、分布式锁、秒杀库存。

### 2.4 异步

1. 当前异步通知是 `@Async + ThreadPoolTaskExecutor`。
2. 不是 MQ。
3. 不要顺嘴说成 Kafka、RabbitMQ。

## 3. 面试官最可能追问的点

## 3.1 为什么要做“先认证、后登录”

建议回答：

1. 这个项目不是开放式用户平台，而是企业内部或半内部业务系统。
2. 只有审核通过的用户才能进入业务页面，能减少无关人员访问业务数据。
3. 登录页本身要先识别当前微信账号状态，再决定是走申请、提示审核中，还是允许微信登录。

追问点：

1. 审核中和未通过有什么区别
2. 为什么要有“资格已收回”和“黑名单”
3. 状态切换后前端怎么处理

## 3.2 登录态怎么做

建议回答：

1. 用户微信登录成功后，后端生成 accessToken，并在数据库里保存 token session。
2. Redis 开启时会把 session 做读取加速缓存，但数据库仍是最终事实来源。
3. 鉴权时优先查 Redis，没命中或 Redis 不可用时回源数据库。
4. Redis 异常时有短暂 bypass，避免缓存问题直接影响登录可用性。

追问点：

1. 为什么数据库是最终事实来源
2. Redis 挂了怎么办
3. 为什么不直接只用 JWT

## 3.3 为什么小程序和后台接口要分开

建议回答：

1. 两端面对的用户类型不同，小程序用户和后台管理员的权限模型不同。
2. 拆成 `/api/v2/miniapp/**` 和 `/api/v2/admin/**` 后，路由鉴权和接口治理更清晰。
3. 后续新增小程序能力时，也能避免把接口写到后台域里。

## 3.4 田块“附近”和“当前匹配”是怎么做的

建议回答：

1. 小程序端先获取当前位置，再把经纬度传给后端。
2. 后端支持按距离排序和范围查询，并给出距离展示文本。
3. 首页会展示附近田块，详情页和录入页会尽量提示“您正处于该田块中”。

追问点：

1. 20km 范围是前端算还是后端算
2. 距离排序怎么做
3. 用户没开定位怎么办

## 3.5 农事记录的动态表单怎么实现

建议回答：

1. 流程步骤可以关联动态表单配置。
2. 后端维护 schema，字段结构会统一校验和归一化。
3. 小程序新增农事记录时，根据步骤加载对应 schema，把扩展数据放到 `extraJson`。
4. 详情展示时再结合 schema 把扩展字段渲染成人可读内容。

追问点：

1. 为什么不用固定表字段
2. schema 怎么防止脏数据
3. 不同步骤怎么保证字段不冲突

## 3.6 种子检测为什么要做幂等

建议回答：

1. 检测记录存在重复点击、弱网重试或前端重复提交的风险。
2. 我们给检测写入增加了 `requestKey`，同一批次下相同 `requestKey` 只允许落一条记录。
3. 服务层会先查已有记录，若并发下仍冲突，再依靠唯一键兜底并回查已有数据返回。

追问点：

1. 为什么用 `(batchId, requestKey)` 做唯一约束
2. 前端 requestKey 怎么来
3. 为什么不用乐观锁

## 3.7 图片上传和审核为什么要单独治理

建议回答：

1. 业务图片不只是文件本身，还涉及审核状态、回收站、资源引用和误删风险。
2. 小程序上传图片后，后台可以审核通过或驳回，驳回资源会进入回收站而不是直接消失。
3. 资源和业务对象之间有引用追踪，删除前能先确认影响范围。

追问点：

1. 文件存哪里
2. 元数据存哪里
3. 为什么要做资源锁

## 3.8 消息通知为什么要异步派发

建议回答：

1. 创建消息任务和真正写所有收件箱记录不是一个量级。
2. 如果同步写，会拖慢审核、发布等主链路。
3. 所以后端先保存任务，再由线程池异步解析收件人并批量写入用户收件箱。

追问点：

1. 异步失败怎么办
2. 为什么不用消息队列
3. 如何知道发送成功还是失败

## 3.9 高德缓存为什么要做双层

建议回答：

1. 地图和天气能力走第三方接口，既有调用额度，也有响应成本。
2. 本地缓存能减少短时间内重复请求的开销，Redis 可以跨实例复用缓存结果。
3. 这样能降低真实远程调用次数，也方便后台统计真实用量和缓存命中率。

追问点：

1. 哪些接口适合缓存
2. TTL 怎么定
3. 缓存失效后会怎样

## 3.10 搜索建议是怎么做的

建议回答：

1. 小程序里的田块和种子批次支持远程搜索建议。
2. 后端会对关键词做归一化，并支持前缀、包含、拼音全拼、首字母和一定的模糊匹配。
3. 这样可以提升搜索体验，尤其适合中文实体词的输入场景。

## 4. 面试时哪些内容不能乱写

1. 不要说你做了微服务拆分。
2. 不要说你实现了 Kafka/MQ 异步链路。
3. 不要说你用 JWT 网关做统一认证。
4. 不要说你做了分布式锁、秒杀、库存扣减。
5. 不要说你做了分库分表和复杂一致性补偿。
6. 不要把 Redis 会话缓存说成了高并发缓存架构。

## 5. 哪些内容可以写，但要看你能不能接住

### 5.1 可以写

1. 小程序认证准入
2. token session + Redis 缓存
3. 高德位置天气接入与缓存
4. 动态表单 schema
5. 种子检测幂等写入
6. 图片上传审核
7. 异步消息派发
8. 搜索建议

### 5.2 前提

1. 你至少要知道这个能力解决了什么问题。
2. 你至少要知道代码里大概怎么实现。
3. 你至少要能说出为什么这样设计，而不是别的方案。

## 6. 建议你准备的项目表达顺序

面试时最好按下面顺序讲，不容易散：

1. 先讲业务背景：这个系统解决什么问题。
2. 再讲系统结构：小程序、后台、后端三端。
3. 再讲主链路：认证准入 -> 田块 -> 农事 -> 种子检测 -> 通知。
4. 最后挑 2 到 3 个技术点展开：比如会话缓存、动态表单、幂等写入。

## 7. 一段比较稳的开场说法

这个项目是一个农业种业业务管理系统，包含微信小程序、后台管理端和 Spring Boot 后端。小程序主要给一线人员用来做认证登录、查看田块、录入农事记录和查询种子批次检测；后台主要做用户审核、田块计划、种子管理和资源审核。后端采用模块化单体架构，比较核心的点有小程序与后台路由隔离、token session + Redis 会话缓存、动态表单扩展、种子检测幂等写入以及高德位置天气接入。

## 8. 如果你被问到“这个项目最能体现你什么能力”

建议回答：

1. 第一是我能把一个真实业务从小程序、后台到后端串起来，而不只是写单个接口。
2. 第二是我对业务状态流比较敏感，比如认证准入、审核状态、记录编辑窗口、资源审核这些都不是纯 CRUD。
3. 第三是我不是只会写页面，也接触了后端的缓存、幂等、异步派发和动态表单这类更偏工程化的能力。
