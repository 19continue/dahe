# 微信小程序项目四个亮点补课

这份文档不是简历稿，而是给你自己补基础和讲项目用的学习稿。目标是把“我知道项目里有这个点”补到“我能解释它为什么这么做、解决什么问题、边界是什么”。

## 1. 先认证、后登录的准入机制

### 这个点本质上在解决什么问题

如果项目让所有拿到微信 `code` 的用户都直接登录，就会出现两个问题：

1. 任何拿到小程序入口的人都能直接进入系统，业务侧没有准入门槛。
2. 审核中、被驳回、被停用、资格被收回的用户，即使之前存在账号，也可能继续拿到登录态。

所以这个项目把流程拆成两段：

1. 先识别微信身份和认证状态。
2. 只有状态满足条件时，才签发 session。

这就是“先认证、后登录”的核心。

### 这个项目是怎么落的

#### 第一步：entry 只查状态，不发 token

- 小程序进入认证页，先调用 `/miniapp/auth/entry`。
- 后端根据微信 `code` 换出 `openid`。
- 通过 `openid` 找用户。
- 如果没用户，返回 guest，提示“先提交认证申请”。
- 如果有用户，但状态是审核中、未通过、停用、资格收回，也只返回状态，不发 session。
- 只有状态已经是 `approved` 且用户可登录，前端才切换到“微信登录”模式。

#### 第二步：apply 负责资料提交，不等于一定登录

- 用户提交认证资料走 `/miniapp/auth/apply`。
- 后端会创建或更新这条小程序用户资料。
- 如果状态还是需要审核，就只返回最新状态，不发 token。
- 如果恰好已经满足可登录条件，才允许发 session。

#### 第三步：wechat-login 才是真正签发 session 的地方

- 用户点微信登录，调用 `/miniapp/auth/wechat-login`。
- 后端再次根据微信 `code` 找用户。
- 再次校验用户当前状态是否允许发 session。
- 只有这一步通过，才创建 `TokenSession`。

### 为什么不在 apply 时无脑发 token

因为“提交资料成功”和“允许进入系统”不是一回事。

- 提交成功，只说明后端收到了资料。
- 允许登录，说明业务规则已经放行。

把这两个动作拆开，准入规则才收得住。

### 面试里你要能说出的关键词

- 微信身份识别和业务准入分离
- 认证状态驱动登录态签发
- guest 态和登录态明确分层
- session 签发前再次校验用户状态

## 2. token session 的 Redis 缓存 + Redis 故障 bypass + 高德位置天气缓存

### 2.1 token session 为什么加 Redis

项目里每次请求鉴权都要根据 `accessToken` 查 `token_session` 表。

如果所有请求都直接打数据库，会有两个问题：

1. 高频鉴权查询会给数据库增加持续读压力。
2. 明明是短生命周期、重复读很多次的数据，却每次都回数据库，成本不划算。

所以这里把 Redis 作为“读取加速层”，但不是唯一存储。

### 数据库和 Redis 的角色分工

- 数据库：最终真相，保证 session 的真实状态。
- Redis：热点读取缓存，减少数据库查询。

这句话你面试时一定要讲稳，因为这是这个实现最重要的边界。

### 查 session 的完整链路

1. 请求进来，拦截器拿到 token。
2. 先看当前 Redis 是否可用。
3. 可用就先查 Redis。
4. Redis 命中则直接返回。
5. Redis 未命中再回源数据库。
6. 数据库查到有效 session 后，再回写 Redis。

### 为什么设计 Redis 故障 bypass

如果 Redis 挂了、超时了、序列化异常了，而你的代码还死磕 Redis，会出现两个问题：

1. 每个请求都要先超时一次，接口整体变慢。
2. Redis 故障会反向拖垮主业务。

所以项目里做了一个很实用的工程手段：

- 一旦 Redis 读写异常，就记录一个 `redisBypassUntilAt`。
- 在这个短时间窗口内，所有 session 查询直接跳过 Redis，回数据库。

这不是完美容灾，但很适合当前项目体量，成本低、收益高。

### 这种 bypass 的优点和代价

#### 优点

- 避免每个请求重复踩 Redis 异常。
- Redis 抖动时主业务还能继续跑。
- 逻辑简单，落地快。

#### 代价

- bypass 窗口里数据库压力会变大。
- 这是“临时降级”，不是根治 Redis 故障。

### 2.2 高德位置和天气为什么要缓存

项目里位置、天气能力来自高德开放接口，这种第三方接口有几个天然问题：

1. 调一次就有网络时延。
2. 有额度和 QPS 限制。
3. 同一位置、同一行政区在短时间内经常重复请求。

所以缓存不是为了炫技，而是为了：

- 减少真实调用次数
- 降低首页和田块页响应时间
- 抗住第三方接口偶发抖动

### 为什么是“本地缓存 + Redis 缓存”

这个实现里缓存策略是有条件的：

- 开 Redis 时，主要用 Redis。
- 关闭 Redis 时，用本地有界缓存兜底。

你可以把它理解成：这是一个“可切换的双层方案”，不是强绑定的复杂多级缓存系统。

### 反向地理编码和天气缓存分别怎么做

#### 反向地理编码

- 输入是经纬度。
- 直接用精确经纬度做 key 命中率太低，所以先做“网格化”。
- 同一小范围内的位置会被折叠到一个缓存 key。
- 这样附近田块、当前位置匹配就能复用结果。

#### 天气

- 天气按 adcode 查。
- 一个行政区在短时间内天气通常不需要秒级更新。
- 所以按 adcode 做缓存，能直接覆盖首页天气快照场景。

### 面试里你要能说出的关键词

- Redis 是 session 读缓存，不是唯一存储
- Redis 故障短暂 bypass，避免故障扩散
- 第三方接口缓存主要为降时延、降调用成本、抗 QPS 限流
- 反向地理编码用网格化 key 提升命中率

## 3. 流程步骤驱动的农事记录录入 + schema + extraJson

### 这个点本质上在解决什么问题

农事记录不是所有步骤都固定字段。

例如：

- 播种要记录品种、播量、播种方式。
- 施肥要记录肥料类型、施肥量。
- 病虫害防治要记录药剂、稀释倍数、喷施时间。

如果把所有字段都硬塞进 `farm_record` 表，会有几个问题：

1. 表会越来越胖。
2. 新增一个业务字段要改表、改后端、改前端。
3. 不同步骤的大量字段长期为空。

所以项目把农事记录拆成两层：

- 固定主干字段：田块、周期、步骤、日期、执行人、天气等。
- 动态扩展字段：放进 `extraJson`。

### schema 在这里扮演什么角色

`extraJson` 不能乱存，否则前端后端都不知道这个 JSON 应该长什么样。

所以系统给每个步骤配一份 `formSchema`，定义：

- 字段 key
- 字段 label
- 字段类型
- 是否必填
- 选项值

这样前端可以按 schema 动态渲染表单，后端可以按 schema 校验 `extraJson`。

### 为什么要“步骤驱动”

不是记录自己决定字段，而是“步骤决定字段”。

也就是：

- 先确定当前农事流程步骤 `stepId`
- 再根据该步骤解析 schema
- 最后按 schema 校验并保存 `extraJson`

这样业务规则有中心，不会出现同类步骤录出的数据结构不一致。

### create / update 的校验流程

1. 前端提交 `stepId` 和 `extraJson`。
2. 后端根据 `stepId` 找到步骤。
3. `StepFormSchemaResolver` 解析出最终 schema。
4. `StepFormSchemaValidator` 校验 `extraJson` 是否符合 schema。
5. 校验通过才允许保存。

### 为什么展示时还要再解析 schema

因为数据库里存的是 `extraJson` 原始键值，例如：

```json
{
  "fertilizerType": "compound",
  "amount": 20
}
```

如果列表和详情页直接把这个 JSON 原样扔给前端，不利于展示。

所以后端又做了一层：

- 把 schema 里的 `key -> label` 解析出来
- 把 select 的 `value -> label` 也解析出来
- 返回 `extraLabelMap`、`extraValueLabelMap`

这样前端就能展示成“肥料类型：复合肥，施肥量：20kg”。

### 为什么说这不是完整低代码平台

因为它当前解决的是“步骤扩展字段”这个场景，不是让任何业务都能无限自定义流程引擎。

你讲的时候要收住，不要吹成“通用动态表单平台”。

### 面试里你要能说出的关键词

- 固定字段 + `extraJson` 扩展字段
- 步骤驱动 schema
- 前端按 schema 渲染，后端按 schema 校验
- 展示层用 label map 把机器字段翻译成人类可读文案

## 4. `@Async + ThreadPoolTaskExecutor` 消息任务异步派发

### 这个点本质上在解决什么问题

用户提交认证、管理员审核、状态变更时，系统要发通知。

如果主接口里同步做这些动作：

- 查收件人
- 组装通知
- 批量写 `user_notice`

那么主接口就会被通知逻辑拖慢。

所以这里做了一个轻量异步：

1. 主请求先写一条 `notice_task`。
2. 然后调用异步方法派发。
3. 异步线程去解析收件人并批量写收件箱。

### 为什么当前不用 MQ

不是不能上，而是当前项目体量下没必要一上来就引入 MQ。

原因很简单：

- 当前通知量级还没有大到必须引入消息中间件。
- `@Async + 线程池` 落地快、维护简单。
- 需求是“不要阻塞主请求”，这个目标它已经能满足。

所以这是一种“先够用、再演进”的方案。

### 线程池在这里的作用

`@Async` 不是自动就安全高效，关键在于给它配专用线程池。

这里单独用了 `authNoticeExecutor`，好处是：

- 不和其他异步任务混用
- 能控制并发度
- 能限制队列长度
- 出问题时日志线程名更清楚

### 派发链路是怎么走的

1. 业务代码创建 `notice_task`
2. 调 `dispatchTaskAsync(taskId)`
3. 异步线程先把任务状态从 `pending/failed` 改成 `sending`
4. 读取任务详情
5. 解析收件人范围
6. 批量写 `user_notice`
7. 更新任务结果：成功数、失败数、状态

这里的“先锁状态再发”很关键，目的是避免同一个任务被重复派发。

### 这种方案的优点和边界

#### 优点

- 主请求响应更快
- 比引入 MQ 简单
- 对当前通知业务足够实用

#### 边界

- 仍然属于单体内异步，不是跨系统最终一致性方案
- 机器挂掉时，没有 MQ 那种天然的持久消息消费模型
- 高峰量再上去时，未来可以演进到 MQ

### 面试里你要能说出的关键词

- 主链路只落任务，异步线程负责扩散
- 专用线程池隔离通知派发
- 先抢占任务状态，避免重复派发
- 当前量级下用 `@Async` 比 MQ 更务实

## 你接下来该怎么补

### 第一轮：把流程背顺

先不要背八股，先把下面四条链路用你自己的话讲顺：

1. 认证页进入到微信登录成功的全链路
2. token 鉴权从拦截器到 Redis 再到数据库的全链路
3. 农事记录从 `stepId` 到 `schema` 再到 `extraJson` 校验的全链路
4. 通知从业务落任务到异步派发完成的全链路

### 第二轮：补设计动机

每个点都补一句“为什么不直接这么做”：

- 为什么不直接登录
- 为什么不只查数据库
- 为什么不把字段全写死在表里
- 为什么不在主线程同步发通知

### 第三轮：补边界和缺点

这一步很重要，因为会让你听起来像真的理解过：

- Redis 只是加速层，不是唯一真相
- bypass 会把压力转回数据库
- `extraJson` 灵活，但复杂查询能力弱
- `@Async` 轻量，但不是 MQ
