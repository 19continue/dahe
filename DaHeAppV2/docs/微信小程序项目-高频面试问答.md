# 大禾种业微信小程序项目高频面试问答

## 1. 这个项目是做什么的

答：
这是一个面向农业种业场景的业务管理系统，包含微信小程序、后台管理端和 Spring Boot 后端。小程序负责认证登录、田块查看、农事记录和种子质量查询，后台负责用户审核、田块与计划管理、种子批次管理、资源审核和消息发布，后端负责鉴权、业务编排、数据存储、缓存、文件上传和地图天气接入。

## 2. 为什么这个项目要做“先认证、后登录”

答：
因为它不是开放式互联网应用，而是企业业务系统。用户只有先通过审核，才能访问田块、农事记录和质量检测等业务数据，所以登录页要先判断当前微信账号状态，再决定是提交申请、提示审核中，还是允许直接微信登录。

## 3. 这个项目后端是什么架构

答：
后端是模块化单体架构，不是微服务。虽然系统包含小程序端和后台端两类接口，但都是由同一个 Spring Boot 服务承接，只是在路由层严格区分了 `/api/v2/miniapp/**` 和 `/api/v2/admin/**`。

## 4. 这个项目的鉴权是怎么做的

答：
项目不是纯 JWT 方案，而是 token session 方案。用户登录成功后，后端生成 accessToken，并在数据库保存 token session；Redis 可用时会把 session 放到 Redis 里做读取加速，鉴权时先读 Redis，失败再回源数据库。权限判断主要由自定义拦截器和路由授权服务负责。

## 5. Redis 在这个项目里具体用在哪

答：
主要用了两个地方。第一是 token session 的读取加速缓存，降低频繁鉴权时对数据库的压力；第二是高德地图位置天气接口缓存，用来减少真实第三方调用次数和额度消耗。

## 6. 为什么 Redis 故障时不能直接让系统不可用

答：
因为 Redis 在这个项目里是加速层，不是唯一事实来源。数据库仍保留完整 session 数据，所以 Redis 异常时应该回退数据库，而不是把用户登录和鉴权链路一起打死。

## 7. 田块“附近”和“当前匹配”是怎么实现的

答：
小程序先获取当前位置，然后把经纬度传给后端。后端支持按距离排序和范围查询，返回距离文本给前端展示。首页会展示附近田块，详情页和录入页还会提示用户是否正处于当前田块中，帮助现场人员更快确认操作对象。

## 8. 农事记录为什么要和流程步骤绑定

答：
因为农事不是无结构的备注，不同步骤关注的信息不同。把记录和流程步骤绑定后，系统可以按步骤展示完成进度，也可以为不同步骤挂载不同动态字段，比如播种和收获关注的参数就不一样。

## 9. 动态表单是怎么做的

答：
后端维护动态表单配置，按模块和场景做 schema 管理。流程步骤、种子批次、种子检测都可以关联对应配置。录入时前端根据 schema 渲染表单，提交时把扩展字段存到 `extraJson`，展示时再结合 schema 做结构化回显。

## 10. 为什么种子检测要做幂等

答：
因为检测记录录入时可能出现重复点击、网络重试或前端重复提交。项目给检测写入增加了 `requestKey`，后端先按 `(batchId, requestKey)` 查询已有记录，如果并发下仍然撞上重复写，再依靠数据库唯一键约束和回查兜底，避免生成重复检测数据。

## 11. 你们为什么没有直接把所有扩展字段都做成固定列

答：
因为农事步骤和种子检测的扩展字段变化比较频繁，如果每次业务调整都改表，会让迭代成本变高。动态表单 + `extraJson` 更适合承载这类变化快、但仍需要基本约束的字段。

## 12. 图片上传是怎么处理的

答：
小程序图片统一走后端上传接口。后端会先做权限校验、内容识别和策略校验，然后把文件落盘并保存资源元数据。小程序上传的图片可以进入待审核状态，后台审核通过后再展示，这样可以避免不合规图片直接进入业务详情页。

## 13. 资源审核为什么不直接删除驳回图片

答：
因为业务上需要可追溯和可恢复。项目里驳回资源不是直接硬删，而是写驳回原因并移入回收站，既减少误删风险，也方便后续复查和恢复。

## 14. 资源引用追踪解决了什么问题

答：
它解决的是“文件看起来没人用，但其实业务上已经引用了”的问题。项目会把资源和田块、农事记录、企业信息、用户头像等业务对象的引用关系持久化记录，删除前可以先看到影响范围，避免误删。

## 15. 消息通知为什么要做成异步任务

答：
因为创建一条消息任务很快，但真正把消息写到所有用户收件箱里可能很重。如果同步做，用户审核、后台发布这些接口会被拖慢。所以项目里先保存 `notice_task`，再通过线程池异步分发到 `user_notice`。

## 16. 为什么这里不用 MQ

答：
当前项目规模下，用 `@Async + ThreadPoolTaskExecutor` 已经能满足消息派发需求，落地成本更低、维护更简单。如果后续消息量继续变大，再考虑引入 MQ 会更合适。

## 17. 高德能力为什么要统一走后端

答：
一是避免在小程序端暴露第三方 key；二是方便统一做缓存、计费、审计和错误处理；三是后台需要查看真实调用量、缓存命中率和额度情况，这些都更适合收口在后端。

## 18. 高德缓存为什么做成“双层缓存”

答：
本地缓存适合处理短时间热点访问，Redis 缓存适合跨实例复用结果。把两层结合起来，既能减少同一进程内重复请求，也能减少不同实例之间的重复远程调用，尤其适合位置和天气这种第三方依赖能力。

## 19. 这个项目里你觉得最能体现工程性的点是什么

答：
我觉得主要有四个点。第一是账号状态流不是简单登录，而是有审核、停用、资格收回、黑名单等完整准入逻辑；第二是后端不是简单 CRUD，而是做了会话缓存、动态表单、幂等写入和异步消息派发；第三是资源上传后还有审核、回收和引用追踪；第四是地图天气能力做了缓存和审计治理，而不是直接前端调第三方接口。

## 20. 如果面试官问“这个项目最复杂的地方是什么”

答：
如果从业务看，最复杂的是账号准入和农事记录链路，因为它既有用户状态，又有流程步骤、动态表单、图片、天气和记录权限策略。如果从后端实现看，我觉得比较值得讲的是会话缓存回退、种子检测幂等和资源治理这几块，因为它们比单纯的增删改查更接近真实业务系统的问题。
