# 微信小程序项目四个亮点定位索引

本文只聚焦你当前准备重点讲的四个亮点，目的是先把“代码在哪、链路怎么走、面试时该讲到哪一层”钉死。

## 1. 先认证、后登录的准入机制

### 后端主链路

- `DaHeServeV2/src/main/java/com/dahe/v2/modules/auth/service/impl/AuthPortalServiceImpl.java`
  - `miniappEntry(...)`
    - 小程序首次进入认证页时调用。
    - 作用是“只查状态，不发 session”。
    - 根据微信 `code` 换 `openid`，查询用户是否存在、是否是管理员账号、当前认证状态是什么。
  - `miniappApply(...)`
    - 用户提交认证资料时调用。
    - 作用是“创建或更新认证资料，并根据状态决定是否允许直接发 session”。
  - `miniappWechatLogin(...)`
    - 已通过审核的用户点击微信登录时调用。
    - 作用是“只给可登录用户签发 session”。
  - `validate(...)`
    - 小程序后续带 token 校验登录态时调用。
    - 不只是校验 token 是否存在，也会再次校验用户当前状态是否仍允许登录。
  - `refreshMiniappSession(...)`
    - 小程序刷新会话时调用。
    - 只允许刷新小程序自己的会话，不允许把后台 token 混到小程序。

### 前端主链路

- `DaHeAppV2/pages/auth/login.vue`
  - `hydrateFromStoredUser()`
    - 页面启动时先用本地用户信息回显状态。
  - `refreshEntryStatus()`
    - 调 `/miniapp/auth/entry`，向后端重新查询当前微信账号的准入状态。
  - `applyEntryResult(resp)`
    - 把后端返回的状态映射为页面的 `login/apply` 两种模式。
  - `submitApply()`
    - 提交认证申请。
  - `submitWechatLogin()`
    - 真正执行微信登录。
  - `formatStatusMessage()` / `resolveStatusBadge()`
    - 把后端状态翻译成用户能看懂的提示文案。

### 支撑策略

- `DaHeServeV2/src/main/java/com/dahe/v2/modules/auth/policy/AuthUserPolicy.java`
  - 定义“什么状态允许发 session、什么状态禁止登录”的规则。
- `DaHeServeV2/src/main/java/com/dahe/v2/modules/auth/domain/AuthMessageCatalog.java`
  - 定义审核中、未通过、停用、资格收回等统一提示文案。

## 2. token session 的 Redis 缓存 + Redis 故障 bypass + 高德位置天气缓存

### token session

- `DaHeServeV2/src/main/java/com/dahe/v2/modules/session/service/impl/TokenSessionServiceImpl.java`
  - `findValidByToken(...)`
    - 先读 Redis，未命中再回源数据库。
  - `cacheSession(...)`
    - 把数据库里的有效 session 写入 Redis。
  - `readSessionCache(...)`
    - 从 Redis 反序列化 session。
  - `deleteSessionCache(...)`
    - token 失效时主动删缓存。
  - `markRedisFailure(...)`
    - Redis 异常后进入短暂 bypass 窗口。
  - `isRedisAvailable()`
    - 决定当前是否还尝试走 Redis。

### 鉴权拦截接入点

- `DaHeServeV2/src/main/java/com/dahe/v2/modules/auth/support/ApiAuthInterceptor.java`
  - 拦截接口请求，从 header/token 中取 `accessToken`，最终调用 `TokenSessionService.findValidByToken(...)`。

### 高德位置和天气

- `DaHeServeV2/src/main/java/com/dahe/v2/modules/amap/service/impl/AmapOpenServiceImpl.java`
  - `reverseGeocodeWithMeta(...)`
    - 根据经纬度反查地理信息，并带回缓存命中来源。
  - `weatherLiveWithMeta(...)`
    - 按 adcode 查询实时天气，并带回缓存命中来源。
  - `readCacheValue(...)`
    - 统一读本地缓存 / Redis 缓存。
  - `putCacheValue(...)`
    - 统一写回本地缓存 / Redis 缓存。
  - `resolveCachePolicy()`
    - 读取缓存策略，决定是否开 Redis、本地缓存容量和 TTL。

### 业务接入点

- `DaHeServeV2/src/main/java/com/dahe/v2/modules/amap/service/impl/AmapOpenApplicationService.java`
  - 把高德基础能力拼成“附近田块、当前位置匹配、首页天气快照”这种业务能力。
- `DaHeAppV2/components/app-home-panel.vue`
  - 首页位置、天气、附近田块展示。
- `DaHeAppV2/pages/field/detail.vue`
  - 田块详情里显示距离和位置匹配信息。

## 3. 流程步骤驱动的农事记录录入 + schema + extraJson

### 主控制器

- `DaHeServeV2/src/main/java/com/dahe/v2/modules/farm/controller/FarmRecordController.java`
  - `create(...)`
    - 新建农事记录前先校验 `stepId + extraJson` 是否匹配。
  - `update(...)`
    - 更新记录时走同样的动态 schema 校验。
  - `validateStepPayload(...)`
    - 入口级校验方法。
  - `resolveStepSchemaMetaMap(...)`
    - 列表/详情展示时解析 schema 元信息。
  - `parseSchemaMeta(...)`
    - 把 schema 转成字段中文 label 和选项 label 映射。
  - `resolveOperatorAssignment(...)`
    - 小程序用户录入农事记录时，执行人强制绑定为自己。

### schema 解析和校验

- `DaHeServeV2/src/main/java/com/dahe/v2/modules/farm/process/support/StepFormSchemaResolver.java`
  - 决定某个步骤最终使用哪份表单 schema。
  - 优先级：动态配置表 > 步骤内置 `formSchema`。
- `DaHeServeV2/src/main/java/com/dahe/v2/modules/farm/process/support/StepFormSchemaValidator.java`
  - 根据 schema 校验 `extraJson`。
  - 当前支持：`required`、数字、日期、时间、日期时间、select、location 等基础类型。

### 前端页面

- `DaHeAppV2/pages/home/record-edit.vue`
  - 真正的动态录入页。
- `DaHeAppV2/pages/field/detail.vue`
  - 详情页会消费后端解析出的 `extraLabelMap`、`extraValueLabelMap`，把 `extraJson` 里的机器字段翻译成可读文案。

## 4. `@Async + ThreadPoolTaskExecutor` 消息任务异步派发

### 线程池配置

- `DaHeServeV2/src/main/java/com/dahe/v2/modules/auth/config/AuthAsyncConfig.java`
  - 定义 `authNoticeExecutor` 线程池。
  - 控制核心线程数、最大线程数、队列长度等。

### 异步派发实现

- `DaHeServeV2/src/main/java/com/dahe/v2/modules/auth/service/impl/NoticeDispatchServiceImpl.java`
  - `dispatchTaskAsync(...)`
    - 从 `notice_task` 取任务，抢占发送状态，异步解析收件人并批量写入 `user_notice`。
  - `resolveRecipients(...)`
    - 根据目标类型解析收件人。
  - `listActiveUserIds(...)`
    - 查全部启用用户。
  - `listAdminIdsByRoute(...)`
    - 查某菜单路由对应的管理员。
  - `listExplicitUserIds(...)`
    - 查显式指定用户。

### 任务发起方

- `DaHeServeV2/src/main/java/com/dahe/v2/modules/auth/service/impl/UserNoticeServiceImpl.java`
  - 业务侧先写 `notice_task`，再调用 `noticeDispatchService.dispatchTaskAsync(taskId)`。
  - 所以整体模式是“主请求只负责落任务，异步线程负责扩散通知”。

## 你面试时最稳的讲法

### 这四个点分别对应什么标签

- 准入机制：认证状态机 + 登录链路收口
- token session：缓存加速 + 故障降级
- 高德位置天气：第三方 API 接入 + 双层缓存
- 农事记录：动态表单 + 业务扩展参数
- 异步通知：削峰解耦 + 主链路降阻塞

### 最好不要讲过头的地方

- 不要把 session Redis 说成“Redis 是唯一 session 存储”，数据库才是最终真相。
- 不要把高德缓存说成“复杂分布式多级缓存平台”，它是面向第三方接口成本和时延的工程优化。
- 不要把农事记录 schema 说成“完整低代码平台”，当前更准确是“步骤驱动的动态字段扩展”。
- 不要把 `@Async` 说成“消息中间件架构”，当前是轻量异步派发，不是 MQ。
