# 大禾种业微信小程序项目介绍

## 1. 项目定位

这是一个面向农业种业场景的业务管理系统，采用前后端分离实现，包含：

1. 微信小程序端：给一线人员使用，完成认证登录、田块查看、农事记录、种子质量查询、通知查看等操作。
2. 后台管理端：给企业管理员使用，完成用户审核、田块管理、流程模板管理、种子批次管理、资源审核、消息发布等治理工作。
3. Spring Boot 后端：统一承接小程序端和后台端接口，负责鉴权、业务编排、数据存储、缓存、文件上传、地图天气接入和审计治理。

项目目标不是做一个展示型小程序，而是把“田块、种植计划、农事记录、种子批次、检测结果、图片证据、位置天气、消息通知”串成一条可落地的业务链路，方便企业后续做追溯、协同和日常管理。

1. `Spring Boot 2.7`
2. `Spring Security`

6. `Redis`

说明：

3. 鉴权主体是“自定义拦截器 + 路由授权服务 + token session”，不是典型 JWT 网关方案。

## 3. 项目描述

该项目是一个服务农业种业场景的业务管理系统，围绕小程序用户认证、田块分布、种植计划、农事记录、种子批次、质量检测、企业介绍、消息通知等业务展开。系统包含微信小程序端、后台管理端和 Spring Boot 后端三部分，通过统一的 `/api/v2/miniapp/**` 与 `/api/v2/admin/**` 路由边界隔离小程序与后台能力，并结合 MySQL、Redis、文件上传、高德地图天气、动态表单配置、资源审核与消息异步派发等能力，支撑一线业务录入与后台治理。

## 4. 核心业务模块

### 4.1 用户认证与准入

1. 小程序账号采用“先认证、后登录”的准入方式。
2. 用户先通过微信获取 code，再走 `/miniapp/auth/entry` 判断状态。
3. 未认证、审核中、未通过、已停用、资格已收回、黑名单等状态都会在登录页做不同展示和限制。
4. 审核通过后，用户才可真正走微信登录，生成 token session 进入业务页。
5. 小程序用户和后台用户完全走不同路由域，避免账号体系混用。

### 4.2 首页看板、定位与天气

1. 首页展示今日记录数、田块总数、种子批次数、最近更新时间等统计信息。
2. 支持获取当前位置，自动展示位置和天气快照。
3. 可根据当前位置展示常用田块、附近田块，并支持下拉刷新。
4. 高德接口调用统一由后端代理，避免小程序端直接暴露 key。

### 4.3 田块分布与种植计划

1. 支持田块分页、详情、搜索、距离排序、面积排序。
2. 支持按省市区乡镇、作物、品种、阶段等条件筛选。
3. 后端通过田块主数据 + 当前种植计划聚合输出结构化作物品种信息。
4. 支持控制台对田块位置、基础信息、封面图和当前计划进行维护。
5. 可按当前位置识别当前田块，并在详情页展示“您正处于该田块中”。

### 4.4 农事流程与农事记录

1. 支持流程模板、流程步骤和步骤排序管理。
2. 农事记录支持分页、详情、分组统计、最近记录时间轴展示。
3. 记录写入时会自动关联田块当前计划、流程步骤和执行人。
4. 步骤支持动态表单 schema，记录额外参数以 `extraJson` 存储。
5. 记录中可绑定图片、天气、湿度、风向、风力、位置等现场信息。
6. 小程序用户的可编辑/可删除能力由“创建者 + 编辑窗口 + 策略配置”共同控制。

### 4.5 种子批次与质量检测

1. 支持种子批次分页、详情、新增、编辑、启停和删除。
2. 支持批次下检测记录的新增、编辑、查询和删除。
3. 检测规则支持样本数配置与发芽率计算。
4. 检测记录增加 `requestKey` 幂等标识，避免重复提交。
5. 批次和检测同样支持动态表单配置，便于扩展业务字段。

### 4.6 通知与控制台

1. 后台支持创建消息任务，再异步派发到个人收件箱。
2. 小程序端支持查看通知、已读、全部已读和删除。
3. 控制台能力通过 `canConsole` 控制，不直接等同于后台角色。
4. 控制台可进入用户审核、田块管理、种子管理等操作页。

### 4.7 企业介绍与分享能力

1. 小程序端支持查看企业概况、产品、荣誉和联系方式。
2. 支持分享小程序给好友，并生成小程序码用于保存和转发。
3. 企业介绍数据由后台维护，小程序端走独立公开接口读取。

### 4.8 资源上传与审核治理

1. 小程序图片上传统一走 `/miniapp/files/upload`。
2. 后端会先做内容识别、策略校验、文件落盘和元数据落库。
3. 小程序上传资源可进入待审核状态，审核通过后再对外展示。
4. 后台支持资源审核、回收站、资源锁、目录治理、静态资源管理。
5. 资源与业务对象的引用关系会被持久化记录，避免误删。

## 5. 后端设计亮点

### 5.1 单体架构下的路由隔离

1. 后端统一采用 `/api/v2/admin/**` 和 `/api/v2/miniapp/**` 两套接口域。
2. 小程序接口必须落在 `modules/miniapp`，后台接口必须走管理端路由。
3. 路由边界在拦截层统一处理，而不是把权限判断散落在业务代码里。

### 5.2 Token 会话与 Redis 缓存

1. 会话数据以数据库为最终事实来源。
2. 当 Redis 可用时，Redis 只做 token session 的读取加速缓存。
3. Redis 出现异常时会进入短暂 bypass，回退数据库，避免缓存故障扩大成登录故障。

### 5.3 高德接口双层缓存与审计

1. 高德位置与天气能力由后端统一代理。
2. 采用“本地有界缓存 + Redis TTL 缓存”降低真实调用次数。
3. 配额、缓存策略、审计保留策略可在后台动态调整。
4. 后台可清理缓存、查看审计趋势、验证 key 和做健康检查。

### 5.4 种子检测幂等处理

1. 检测写入支持 `requestKey`。
2. 先按 `(batchId, requestKey)` 查询，命中则直接返回已有记录。
3. 并发冲突下若数据库唯一键异常，再回查已有记录返回，避免重复落库。

### 5.5 动态表单配置治理

1. 动态表单按“模块 + 场景 + 版本”管理。
2. schema 会做字段类型白名单、key 自动清洗、options 归一化和去重。
3. 删除配置前会检查是否已被农事步骤、种子批次或检测记录引用。

### 5.6 资源治理与文件上传编排

1. 上传逻辑集中在 `MediaAssetUploadFacadeServiceImpl`，降低控制器复杂度。
2. 上传前做权限校验、内容识别、策略限制、目录锁定和文件落盘。
3. 小程序上传默认进入对应资源目录，后台可做审核、回收和永久删除。

### 5.7 异步通知派发

1. 消息发布和真正写收件箱拆成两段。
2. 后台先写 `notice_task`，再由 `@Async("authNoticeExecutor")` 异步解析收件人并批量写入 `user_notice`。
3. 这样可以避免用户审核、消息发布等操作被大批量收件箱写入阻塞。

### 5.8 小程序搜索建议

1. 对田块和种子批次提供远程搜索建议。
2. 支持关键词前缀、包含、拼音全拼、首字母和一定程度的模糊匹配。
3. 既提升列表页搜索体验，也能体现后端对搜索索引的额外设计。

## 6. 如果写成“个人负责内容”，建议这样组织

2. 负责小程序准入流程实现，梳理“未申请、审核中、未通过、已通过、已停用、资格已收回、黑名单”等状态流，并完成小程序登录页与后台审核页联动。
3. 负责田块与农事记录链路开发，支持当前位置匹配田块、流程步骤驱动的农事录入、动态表单参数回填和时间轴展示。
4. 负责种子批次与质量检测模块开发，补齐检测记录幂等写入、样本数计算和动态字段扩展能力。

6. 参与高德位置天气能力接入和缓存治理，支持首页位置天气展示、附近田块推荐和后台审计查看。

## 7. 项目亮点总结

1. 不是单一展示型小程序，而是包含小程序端、后台端和后端服务的完整业务系统。
2. 业务链路完整，从“用户准入”一直覆盖到“田块、记录、批次、检测、通知、资源治理”。
3. 单体项目里做了比较清晰的模块边界和端侧路由隔离，结构上比普通 CRUD 项目更规整。
4. 有真实的缓存、异步派发、幂等、动态表单、资源审核、地图天气接入等可讲点。
5. 更适合包装成“农业种业业务管理系统”或“农业现场数据采集与追溯系统”，而不是简单说“我写了个小程序”。

## 8. 写项目介绍时的注意事项

1. 不要写成微服务项目。当前仓库是模块化单体，不是 Spring Cloud 微服务。
2. 不要写 Kafka、MQ、分库分表、分布式锁。当前项目没有这些真实落地。
3. 不要把鉴权写成 JWT 网关方案。当前更准确的是“token session + Redis 会话缓存 + 自定义鉴权拦截”。
4. 不要把 Redis 写成项目全链路缓存。当前主要是会话缓存和高德缓存。
5. 如果面试官深问某个模块，只讲你真实看过、改过、能解释的部分，不要因为项目里有代码就默认自己都能讲。
