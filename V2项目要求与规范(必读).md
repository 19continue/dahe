1. 文件读写统一使用 `UTF-8` 编码（无 BOM），主语言为简体中文。
2. 数据库变更由你执行，别让本人动手。后端数据库一张表维护一个 SQL 文件，方便日后部署工作。
3. 每次功能改动，都需要提交本地 git 仓库，提交到 `main` 分支，并在提交时说清楚改了什么。在 `git.md` 中记录每次的提交。
4. 本项目企业需要实际使用，设计时一定要考虑完整并避免漏洞。代码带详细注释，注释讲得清晰。注重代码规范，代码要整洁。代码要注重设计和解耦，产出好迭代、好维护的代码。
5. 每开启一轮新的任务，清空 `todo.md`，然后将任务整理记录在 `todo.md` 中并时刻追踪进度，以防止对话中断。
后端：
    1. 模块目录结构必要时才能开启新的目录，禁止随意新开多层目录；业务编排代码统一放在模块既有 `service` 体系（`service`/`service/impl`）中，一些常用的可以写在工具类中，避免目录膨胀和后续维护困难。
    2. 新增模块必须先定义清晰边界：后台 `admin` 与小程序 `miniapp` 接口严格分离，非 `miniapp` 模块禁止新增小程序接口。
    3. 鉴权必须由 `auth` 模块统一承载（Spring Security + 统一策略），业务模块不写散落式权限判断；`controller` 只做入参校验与响应组装，业务逻辑必须放在 `service`。
    4. 菜单能力统一使用“路径码/路由键”作为权限标识：后端仅返回路径码，前端从本地路由读取中文标题并与后端权限码求交集生成可见菜单。
    5. 新增模块或 API 变更时，必须同步维护 API 迁移文档（旧接口、新接口、迁移状态、前端改造状态、下线时间）。
    6. 错误提示必须可读：鉴权失败、越权、参数错误要返回明确业务提示，禁止对前端只返回“系统内部错误”。
    7. 新增模块交付前必须完成：接口自测、关键鉴权路径验证、文档更新（模块解读、待改进点、后端总文档、API 迁移清单）。
小程序：
    1.凡是组件都优先使用uniapp版的tdesign组件，使用时通过context7查询文档，样式上若涉及主题色可以修改，其他可以使用原生样式。
    2.所有请求非特殊要求不要一次请求特别大量的数据，能一次请求解决的不要请求多次。
